## 1. 这台服务器有 GPU 吗？

当前没有。根据服务器探查结果，`node01` 和 `node02` 均未检测到 GPU 设备，也没有配置 Slurm GPU 资源。因此不要在作业脚本中使用 `--gres=gpu:1`，也不要提交到不存在的 `gpu` 分区。

## 2. 为什么不要在登录节点直接运行大任务？

`node01` 同时承担登录节点、Slurm 控制节点和计算节点角色。登录节点主要用于登录、编辑代码、编译、小规模测试和提交作业。如果在登录会话中直接运行大规模计算，可能影响其他用户登录、Slurm 服务和文件操作。

大型任务建议通过 `sbatch` 提交。

## 3. 任务一直是 PD 怎么办？

可以先查看原因：

```bash
squeue -u $USER
scontrol show job JOBID
```

常见原因包括资源不足、队列限制、账户限制、节点故障、作业脚本错误等。

## 4. 怎样判断任务是否正常运行？

建议查看：

```bash
squeue -u $USER
tail -f JOBID.log
tail -f JOBID.err
```

如果日志长时间没有输出，或出现 `nan`、`segmentation fault`、`out of memory` 等信息，需要进一步检查。

## 5. 程序显示内存不足怎么办？

每台节点约 377 GiB 内存，但程序仍可能因为一次性读入过大数组或多进程重复占用内存而失败。可以考虑：

- 减小输入规模；
- 分块处理数据；
- 减少并行进程数；
- 检查程序是否为每个进程复制了完整大数组；
- 将中间数据写入磁盘而不是全部放入内存；
- 通过 `free -h`、`top`、`sacct` 等命令检查实际内存占用。

## 6. `/data1` 和 `/data2` 应该怎么用？

`/data1` 和 `/data2` 是主要数据目录，适合存放输入文件、输出结果和中间数据。建议每个用户建立自己的项目目录，例如：

```text
/data1/$USER/project_name/
├── input/
├── scripts/
├── logs/
└── results/
```

不建议把大型计算数据长期放在系统盘或混杂在 home 目录中。

## 7. 多节点任务性能为什么可能不理想？

当前节点间为 1 GbE 以太网互联，未检测到 Infiniband。对于单节点 CPU 任务影响较小，但对强通信 MPI、多节点频繁同步、大规模并行 I/O 任务可能形成瓶颈。

如果多节点任务扩展性不好，建议先比较：

- 1 节点 64 核；
- 1 节点 128 核；
- 1 节点 256 核；
- 2 节点相同总核数。

通过对比判断瓶颈来自计算、通信还是 I/O。

## 8. 如何向管理员反馈问题？

建议提供以下信息：

- 用户名；
- 作业 ID；
- 提交脚本；
- `.log` 和 `.err` 文件；
- `squeue -u $USER` 截图；
- `scontrol show job JOBID` 输出；
- `sacct -j JOBID --format=JobID,JobName,State,ExitCode,Elapsed,AllocCPUS,ReqMem,MaxRSS` 输出；
- 问题发生的大概时间。

## 9. 管理员后续可以优先改进什么？

从使用体验和稳定性角度，建议优先关注：

| 优先级 | 建议 | 说明 |
|---|---|---|
| 高 | 设置正式分区和时间限制 | 当前 `debug` 分区时间无限制，长期使用容易造成资源占用不均 |
| 高 | 明确用户使用规则 | 例如每人最多使用核数、长任务申请流程、数据清理周期 |
| 中 | 统一两节点内核版本 | 减少环境差异带来的潜在问题 |
| 中 | 规划共享存储使用方式 | 减少大规模 NFS I/O 对任务的影响 |
| 中 | 评估网络升级 | 若多节点 MPI 或并行 I/O 需求增加，可考虑 10/25/100 GbE 或 Infiniband |
| 低 | 补充公共软件环境 | 固化编译器、MPI、Python、HDF5、FFTW 等常用环境 |
